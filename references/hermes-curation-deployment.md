# Hermes Stack curation deployment

`run-stack-bookmark-curation.sh` separates `collection` and `curation` with independent local locks. Collection may be scheduled nightly; curation is review-only and must not activate or publish capabilities.

The installer is dry-run by default. It prints the exact `hermes cron create '17 1 * * *' --name stack-bookmark-collection --script ~/.hermes/scripts/stack-bookmark-collection.sh --no-agent` and weekly-curation equivalents. It creates separate thin wrappers because Hermes cron script jobs do not accept script arguments. `--install-wrappers` installs those wrappers without creating jobs so they can be run and inspected first. Each wrapper supports `--dry-run` for a non-advancing manual check and `--manual` for the required apply-mode run-now proof; its scheduled/default path also uses apply mode so successful observations advance exactly once without activating or publishing a capability. A later enablement requires existing wrappers, `--enable`, the explicit `--approval-token I_APPROVE_HERMES_STACK_CURATION`, and the exact positive status marker `Gateway is running` (never `Gateway is not running`). It also requires an explicit owner-only (`0700`) verification receipt directory (`STACK_HERMES_VERIFICATION_DIR`, default `~/.local/state/stack/receipts`) containing recent owner-only (`0600`) successful manual collection and curation receipts. Each receipt must be complete, have matching `receipt_type`/`phase`, state `manual_run: true`, and record `mode: apply`; stale, partial, dry-run-only, malformed, or scheduler-only receipts fail closed. The installer separately checks the Stack sources, fetch policy, catalog, and curation runner; it does not infer readiness from the old `STACK_BOOKMARK_CANDIDATES` bridge. The normal deployment proof is wrapper installation, manual dry runs, successful manual apply-mode run-now receipts, then explicit enablement.

Receipts are owner-local under `~/.local/state/stack/receipts/`. Collection, curation, partial source failures, scheduler lock failures, and publication outcomes remain distinct; no receipt is proof of publication unless it carries Stack's source commit, registry digest, target results, and verification time.
