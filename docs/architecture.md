# Stack architecture

## Product boundary

Stack curates capabilities used to design and build software. A capability can
be active only when it materially improves product planning, interface design,
implementation, review, verification, or shipping.

**Inclusion test:** name the concrete design/build workflow improved, the
repeatable skill or reference knowledge supplied, and the evidence that it
helps. Orchestration qualifies only when it directly supports that named
workflow. Personal operations, finance, household administration, shopping,
file organization, and general knowledge management fail the test unless they
provide that direct support.

## Ownership map

| Surface | Owner | Contract |
| --- | --- | --- |
| `skills/**/capability.json` | Stack | Authoritative capability-local manifest: purpose, domain, artifact type, lifecycle, provenance, validation, overlap, supported runtimes, and publish targets. |
| `registry/capabilities.json` | Stack compiler | Deterministic aggregate catalog generated by `scripts/build-capability-registry.py`; never edit it directly. |
| `scripts/audit-capabilities.py` and `artifacts/audits/` | Stack | Read-only estate evidence and proposed disposition; review makes consequential decisions. |
| `scripts/collect-bookmark-candidates.py` | Stack | Read-only source collection into an owner-local ledger; source capture is not completion. |
| `scripts/triage-bookmark-candidates.py`, `scripts/prepare-capability-candidate.py`, and `scripts/evaluate-capability-candidate.py` | Stack | Redacted candidate packets, evaluation preparation, and fixed-policy isolated evaluation; never activation or publication. |
| `scripts/compile-runtime.py` and `scripts/install-runtime.py` | Stack | Catalog-driven staging, atomic target switching, receipts, and rollback. |
| Codex, Claude, Hermes | Runtime/deployment surfaces | Consume generated outputs and verify discovery; they do not own a competing catalog or raw-worktree installation. |
| `docs/private-overlay.md` | Owner-local deployment | Defines the optional private boundary; private payloads never enter public Stack output. |

## Catalog and compiler

Capability-local manifests are versioned truth. The catalog compiler validates
them and creates the aggregate. Runtime compilation consumes the aggregate and
selects only `active` entries compatible with a declared runtime target. A
generated runtime manifest binds each output to the Stack source commit and
registry digest; it is evidence of a particular publication, not another
editable registry.

```sh
python3 scripts/build-capability-registry.py
python3 scripts/build-capability-registry.py --check
```

Seeded `candidate` entries with pending audit status are inventory, not
approval. A catalog entry becoming `active` is still intermediate: target
compilation, installation/discovery, and a receipt must succeed before calling
it published.

## Evidence and review

The audit records tracked artifacts, callable entrypoints, nested entrypoints,
provenance, validation gaps, overlaps, and a proposed disposition. Batch review
can confirm mechanically unchanged inventory and keeps after calibration.
Merge, demote, move, archive, hold, activation, and publication require
item-level human review. Scores inform review; they do not make decisions.

Use the audit command to create a durable evidence packet:

```sh
python3 scripts/audit-capabilities.py --output-dir /tmp/stack-capability-audit
```

No audit command moves, deletes, activates, or publishes a capability.
